Chapter 10: Debugging and Troubleshooting
Errors, Exceptions, and Warnings:
System error ― there’s a problem with the system or external devices with which the program is interacting.
Programmer error ― the program contains incorrect syntax or faulty logic; it could even be as simple as a typo.
User error ― the user has entered data incorrectly, which the program is unable to handle
An exception is an error that produces a return value that can then be used by the program to deal with the error
Stack Trace is a sequence of functions or methods that lead to the point where the error occurred
A warning can occur if there’s an error in the code that isn't enough to cause the program to crash
Strict Mode:
Strict mode is a feature that produces more exceptions and warnings and prohibits the use of some deprecated features.
To enable strict mode, simply add:
'use strict';
You can also use strict mode on a per function basis if you add the use strict within the function rather than globally.
Modules are in strict mode by default
Linting Tools (JS Lint, JS Hint, ES Lint)
Even more ‘strict’ than strict mode. Linting tools will identify sloppy programming practices or syntax errors. Will also
ensure that style conventions are met.
Feature Detection:
This is to ensure that the browser being used supports any APIs that you might be using.
Debugging:
Use alert() to show dialog at certain points to verify functionality
You can also use console.log() for the same results
Use breakpoints to step through the code to watch it work (you can enter debugger; directly into the js and this will force
the browser to create a breakpoint at that line) REMEMBER to remove debugger; from finished project
Error() object can be used to create an error.
EvalError is not used in the current ECMAScript specification and only retained for backwards compatibility. It was used to
identify errors when using the global eval() function.
RangeError is thrown when a number is outside an allowable range of values.
ReferenceError is thrown when a reference is made to an item that doesn’t exist. For example, calling a function that hasn't been defined.
SyntaxError is thrown when there’s an error in the code’s syntax.
TypeError is thrown when there’s an error in the type of value used; for example, a string is used when a number is expected.
URIError is thrown when there’s a problem encoding or decoding the URI.
InternalError is a non-standard error that is thrown when an error occurs in the JavaScript engine. A common cause of this too much recursion.
Throwing Exceptions:
Using the throw statement you can throw your own exceptions (basically creating a system to identify your own errors)
It is best practice, however, to throw an error object. This can then be caught in a catch block
Use the try catch and finally blocks to handle excetions.
Try- will run the code inside the block as normal, but if an exception occurs it will pass the error object that is thrown onto a catch block
Catch-This will handle the error if the try block catches an exception
Finally- this will always be executed after the try and catch blocks regardless of whether they caught an exception or not
Tests- simple functions created to test other functions. This well help with debugging purposes as well
Test-driven development(TDD) is the process of writing tests before any actual code.
1.Write tests (that initially fail)
2.Write code to pass the tests
3.Refactor the code
4.Test refactored code
5.Write more tests for new features
You usually keep testing code in a separate file from the js code
